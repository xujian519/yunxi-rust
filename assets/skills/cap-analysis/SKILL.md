## 【核心强化】三级技术分析框架

```
一级：表面特征提取  →  二级：技术手段解构  →  三级：技术效果对比
       (20%深度)          (60%深度)         (100%深度)
```

**三级分析的递进关系**：
- **一级**：识别有什么技术特征
- **二级**：理解技术手段如何工作
- **三级**：对比技术效果的实现机制和差异

```markdown
## 层次2：功能区别（中等深度）

## 层次3：原理区别（深度）
├─ 工作原理不同: {{实现功能的原理不同}}
├─ 作用机理不同: {{产生效果的根本机制不同}}
└─ 技术思路不同: {{解决问题的根本思路不同}}

## 层次4：技术启示判断（关键）
┌────────────────────────────────────────────────────────┐
│ 对于区别特征X，判断D2是否给出技术启示：                │
├────────────────────────────────────────────────────────┤
│ D2公开的内容: {{D2如何公开类似特征}}                      │
│ 技术领域差异: {{D2的领域与本申请的领域差异}}               │
│ 技术问题差异: {{D2解决的问题与本申请的问题差异}}           │
│ 技术效果差异: {{D2达到的效果与本申请的效果差异}}           │
├────────────────────────────────────────────────────────┤
│ 技术启示判断:                                          │
│ ☐ 有明确启示 (D2直接教导将特征用于本领域)               │
│ ☐ 有隐含启示 (本领域技术人员容易想到)                   │
│ ☐ 启示不明显 (需要创造性劳动才能结合)                   │
│ ☐ 无技术启示 (技术领域/问题/效果差异大)                  │
└────────────────────────────────────────────────────────┘

**判断依据**：
- 技术领域相同或相近 + 技术问题相同 + 技术效果相同 → 有启示
- 技术领域不同 OR 技术问题不同 OR 技术效果显著不同 → 启示弱
- 需要克服技术障碍才能结合 → 无启示
```

## 【使用说明】

### 适用场景

1. **意见答复（Task 2.2）**：强制使用本提示词
2. **无效分析（CAP07）**：强烈建议使用
3. **技术对比（CAP02）**：深度对比时使用

### 执行顺序

```
步骤1：解析审查意见 → 识别驳回理由
步骤2：技术解构 → 目标专利 + 对比文件
步骤3：深度对比 → 三级对比矩阵
步骤4：区别特征 → 本质识别 + 技术启示判断
步骤5：策略制定 → HITL确认
步骤6：答复撰写 → 最终确认
```
